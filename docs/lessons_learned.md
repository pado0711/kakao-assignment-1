# Lessons Learned

## 동일 출처 검증은 실제 Host를 기준으로 한다

Next.js 개발 서버나 배포 프록시는 내부 URL의 host를 정규화할 수 있다. `request.nextUrl.origin`과 Origin 문자열을 직접 비교하면 정상 요청도 차단될 수 있다.

- Origin URL의 `host`를 `x-forwarded-host` 또는 `host` 헤더와 비교한다.
- 파싱할 수 없는 Origin은 거부한다.
- 프록시 변경 후에는 실제 HTTP 요청으로 mutation을 확인한다.

## 반복 일정은 규칙과 occurrence 예외를 분리한다

무기한 반복을 날짜별 Todo 행으로 생성하면 데이터가 끝없이 증가한다. 반복 규칙은 한 행으로 저장하고 특정 날짜의 완료·수정·삭제만 예외 테이블에 기록한다.

- 규칙 변경으로 더 이상 발생하지 않는 날짜의 예외는 제거한다.
- occurrence API는 해당 날짜가 실제 반복 날짜인지 서버에서 검증한다.
- 전체 반복 삭제 시 예외도 외래키 cascade로 삭제한다.

## 사용자 소유권은 모든 리소스 조회 조건에 포함한다

리소스를 ID만으로 조회한 후 별도로 사용자 검사를 하면 누락되기 쉽다.

- `resource.id`와 `resource.user_id`를 한 쿼리에서 함께 검사한다.
- 다른 사용자의 리소스도 404로 처리해 존재 여부를 숨긴다.
- 테스트에서 두 사용자를 만들고 교차 수정·삭제를 시도한다.

## 브라우저 토큰 저장을 피한다

세션 원문은 Next.js의 HttpOnly 쿠키에만 저장하고 FastAPI DB에는 해시만 저장한다. Client Component는 토큰이 아니라 동일 출처 `/api`만 호출한다.

## 자동 검사와 런타임 스모크 테스트를 함께 수행한다

타입 검사와 빌드가 통과해도 프록시 host, 쿠키, Origin처럼 실행 환경에서만 드러나는 문제가 있다.

- 백엔드 API 테스트로 도메인·권한 규칙을 검증한다.
- 컴포넌트 테스트로 입력·요일 선택·Modal 경로를 검증한다.
- 최종 단계에서 회원가입 → 생성 → 서버 렌더링 흐름을 실제 HTTP로 확인한다.
